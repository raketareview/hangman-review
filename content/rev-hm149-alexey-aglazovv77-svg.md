## НЕДОСТАТКИ РЕАЛИЗАЦИИ

1. Юзеру не нужна эта информация, которая рапечатывается на старте.  
Такого адреса у меня на компьютере нет
```
C:\Users\aglaz\eclipse-workspace\HangMan1\words.txt
```

2. Не печатаются картинки
```java
У вас 9 попыток отгадать слово
Введите букву: 
ы
Текущее слово: ******




=========
Неверно! Ошибки: [Ы]
```
Возможно, картинка напечаталсь, просто она пустая.  
Но для пользователя игры пустая картинка = отсутствие картинки.  
Если пустая картинка означает пустую виселицу, то нарисуй пустую виселицу.

3. Команды работают неправильно
```
Хотите сыграть еще? y/n
xochy
Игра закончена!
```
Программа должна реагировать только на заявленные команды.  
Все остальные команды она должна считать ошибочными и игнорировать.

## ЗАМЕЧАНИЯ

- Константы должны быть написаны стилем UPPER_SNAKE
```java
private static final String[] STAGESHANGMAN 

//ПРАВИЛЬНО:
private static final String[] HANGMAN_STAGES
```

**1. Нейминг**

- Переменным давай осмысленные названия. Однобуквенные имена в большинстве случаев уместны только в качестве имен индексов в циклах
```java
s = sc.nextLine().toUpperCase();
if (s.equals("E")) {...}

//ПРАВИЛЬНО:
String command = scanner.nextLine().toUpperCase();
if (command.equals("E")) {...}
```

- Перечни в виде массивов, списков, сетов и т.п. должны называться во множественном числе
```java
int[] stageIndex

//ПРАВИЛЬНО:
int[] stageIndexes
```

- Методу неважно, является ли принимаемое им в аргументы слово случайным или нет- он с любым словом внутри себя работает одинаково.  
Название принимаемого аргумента должно объяснять, что хранит аргумент, а не историю его возникновения
```java
boolean openLetter(String randomWord, char[] masked, char letter) 

//ПРАВИЛЬНО:
boolean openLetter(String word, char[] masked, char letter) 
```
*Oracle Java code conventions, part."Naming conventions"*

**2. Нарушение конвенции кода**

- Константы должны стоять выше всех остальных полей.
- Поля классса не должны стоять между методами.
- Если в классе есть точка входа main, то этот метод должен стоять выше всех остальных методов.

**3. Форматирование строк**

Если нужно печатать или создавать строку с более, чем одним подстановочным значением или значение вставляется внутрь сообщения, используй форматирование- тогда сразу будет виден весь шаблон
```java
System.out.println("Верно! Осталось " + attemptCount + " " + attempt + " отгадать слово");

//ПРАВИЛЬНО:
System.out.printf("Верно! Осталось %d %s отгадать слово %n", attemptCount, attempt);
```

**4. Нарушение DRY**, магические буквы, числа, слова. Вводи константы 
```java
System.out.printf("Игра ВИСЕЛИЦА \nPush [N]ew game or [E]xit\n");
if (s.equals("E")) {...}
if (!s.equals("N")) {...}
{...} while (!s.equals("N"));

//ПРАВИЛЬНО:
private final static String START = "N";
private final static String QUIT = "E"; 

System.out.printf("Игра ВИСЕЛИЦА \nPush [%s]ew game or [%s]xit  \n", START, QUIT);
if (s.equals(QUIT)) {...}
if (!s.equals(START)) {...}
{...} while (!s.equals(START));
```

**5. Нарушение DRY, дублирование логики и текстовых сообщений**

Признание не освобождает от ответственности
```java
String attempt = switch (attemptCount) {
  case 1 -> "попытка";
  case 2, 3, 4 -> "попытки";
  default -> "попыток";
}
System.out.println("Осталось " + attemptCount + " " + attempt + " отгадать слово");
//....

// понимаю что код задвоен(из-за перехода от 5 к 4, при attempt = 4, падеж был от 5), не смог реализовать иначе.
// switch выносил из блока if параллельно выносил счетчик из метода, но все равно работало некорректно и метод получался слепым
String attempt = switch (attemptCount) {
   case 1 -> "попытка";
   case 2, 3, 4 -> "попытки";
   default -> "попыток";
};
System.out.println("Верно! Осталось " + attemptCount + " " + attempt + " отгадать слово");
System.out.println("ошибки: " + errors);
```

Общий код выноси во вспомогательные методы:
```java
printAttemptMessage(attemptCount);
//...

System.out.print("Верно! ");
printAttemptMessage(attemptCount);
System.out.println("ошибки: " + errors);
//...

private static void printAttemptMessage(int attemptCount) {
```

**6. Нарушение DRY. Магические буквы, числа, слова выноси в константы**
```java
File file = new File("C:\\Users\\aglaz\\eclipse-workspace\\HangMan1\\words.txt");

//ПРАВИЛЬНО:
private final static String FILE_PATH = "C:\\Users\\aglaz\\eclipse-workspace\\HangMan1\\words.txt";

//...

```

Кстати, этот путь абсолютный и он будет корректным только на одном компьютере в мире- твоем. У меня такого адреса нет
```java

```
Не используй абсолютные адреса. Если нужно показать полный путь, высчитывай его на основе относительного пути.

**7. Не печатай сообщений, которые зависят от падежей**
```java

  case 1 -> "попытка";
  case 2, 3, 4 -> "попытки";
  default -> "попыток";
};
```
Меняй формулировки сообщений таким образом, чтобы не зависеть от падежей:

**8. Не создавай строки через new**

Не создавай строки не через `new`- это связано с особеностями устройства класса String.  
Создавай строки через специально предусмотренные для этого методы
```java
System.out.println("Текущее слово: " + new String(masked));

//ПРАВИЛЬНО:
System.out.println("Текущее слово: " + String.valueOf(masked));
```

**9. Компилируй регулярные выражения**

Если регулярное выражение используется многократно, как здесь, то нужно его откомпилировать.  
Компиляция регулярного выражения в объект Pattern с помощью Pattern.compile() позволяет повысить производительность при многократном использовании
```java
if (!word.matches("[А-Яа-яЁё]+")) {...}

//ПРАВИЛЬНО:
private static final String REGEX = "[А-Яа-яЁё]+";
private static final Pattern PATTERN = Pattern.compile(REGEX);

//...
Matcher matcher = PATTERN.matcher(word);
if(!matcher.matches()) {...}
```
Блох пишет:
```java
"Хотя String.matches — простейший способ проверки, соответствует ли строка регулярному выражению, 
он не подходит для многократного использования в ситуациях, критичных в смысле производительности."
```
Здесь, конечно, нет критической ситуации с производительностью.  
Но здесь нет и необходимости просто так использовать неоптимальный алгоритм при работе с регулярными выражениями.  
*Блох "Java. Эффективное программирование", изд.3, гл.2.6*

**10. Реакция на ошибки** 

Если по указанному адресу не будет найден файл со словами, то программа начнет работать некорректно, потом аварийно вылетит и напечатает непонятные для юзера сообщения в консоль
```java
Файл не содержит слов.
Ошибка чтения файла: words.txt (Не удается найти указанный файл)
-----------------------------------
Игра ВИСЕЛИЦА 
Push [N]ew game or [E]xit
-----------------------------------
n
Exception in thread "main" java.lang.IllegalArgumentException: bound must be positive
	at java.base/java.util.Random.nextInt(Random.java:551)
```
Здесь должна быть другая реакция на отсутствие файла со словами.  
Нужно сказать юзеру, что файл со словами по указанному пути открыть не удалось и поэтому работа программы будет завершена. 

Причем, путь нужно указать не относительный, как у тебя: `words.txt`.  
А абсолютный, чтобы юзер знал, где конкретно ему нужно искать файл: `c:/..../words.txt`.

После этого корректно завершить работу программы без вываливания красных эксепшенов в консоль.

**11. Чтение слов из файла** 

Чтение слов из файла стоит вынести в отдельный утилитный класс.  
Процедурный стиль программирования разрешает создавать и использовать пользовательские классы.

**12. Неправильное использование типов данных**

Массивы нужны для хранения группы значений. Для хранения одного значения используй одиночную переменную
```java
int[] stageIndex = {0};
//...
stageIndex[0]++;

//ПРАВИЛЬНО:
int stageIndex = 0;
//...
stageIndex++;
```

**13. Передача аргументов в методы**

Если одни и те же данные используются в разных методах класса, то эти данные нужно не перекидывать туда-сюда между методами, а сделать их полями класса
```java
private static boolean openLetter(String randomWord, char[] masked, char letter)  {...}

//ЗДЕСЬ ЛУЧШЕ:
private static String word;  //эти данные нужны на протяжении всего игрового процесса
private static char[] masked;

private static boolean openLetter(char letter)  {...}
```
Да, вопрос о том, в каких случаях и когда поля должны браться из окружения, а когда передаваться между методами, дискуссионен и не так прост.
Об этом нужно думать отдельно в каждом конкретном случае.

**14. Распечатка картинок виселицы**

Картинки хранятся в статическом массиве, это хорошо.  
Картинки и метод их распечатки нужно вынести в отдельный утилитный класс `Renderer` и его метод `void render(int stageIndex)`.  
Это разгрузит классы с логикой от хранения графики.

Программа в процедурном стиле может состоять из нескольких классов.  
В этом случае классы выполняют ту же роль, что и модули в процедурных языках.  
То есть, являются контейнерами процедур.

**15. Комментарии**

Не делай комментарии ради комментариев
```java

  """
	   +---+
		 |   |
		 |   O
		 |
		 |

	""",
```
Когда в проекте много каментов, это плохо- пользы от них практически нет, они только забивают пространство и мешают читать код.  
В идеале, комментариев вообще не должно быть(кроме `TODO`), код должен объяснять сам себя через правильный нейминг и лаконичный код.

**18. Не бросай исключения в космос**

Метод main- последняя точка в программе, после которой исключение аварийно закрывает работу программы.  
Если ты знаешь, что в программе могут возникнуть исключения, то обработай их внутри программы
```java
public static void main(String[] args) throws IOException
```
Хотя бы просто перехвати, напечатай сообщение "Критическая ошибка, работа программы будет прекращена" и корректно закрой программу.  
Сообщение без конкретики- не самый лучший вариант. Но это лучше, чем если программа вылетит с красными эксепшенами и непонятными сообщениями.

## АРХИТЕКТУРА

Код тяжело читается из-за божественного метода main.  

Если какой-то метод получается ощутимо длинным, то есть больше 20-30 строк, то скорее всего, он делает много разных дел одновременно.  
Такой метод нужно сокращать: выявить в нем отдельные смысловые блоки и вынести их во вспомогательные методы.

Например, смотрим на `main()` и, среди прочего, видим в нем отдельные смысловые блоки 
```java
public static void main(String[] args) throws IOException {
  //много строк кода
  System.out.println("-----------------------------------");  <-- БЛОК #1
  System.out.printf("Игра ВИСЕЛИЦА \nPush [N]ew game or [E]xit\n");
  System.out.println("-----------------------------------");

  String s = "";   
  do {  <-- БЛОК #2
    s = sc.nextLine().toUpperCase();
    if (s.equals("E")) {
    System.out.println("Вы вышли из игры");
      return;
    } else if (!s.equals("N")) {
      System.out.println("Вы вводите неверное значение!!!");
    }
  } while (!s.equals("N"));
  //еще много строк кода
}
```

Выносим логику этих блоков во вспомогательные методы:
```java
public static void main(String[] args) {
  //...
  printGreeting();

  String command = inputCommand();
  if(isQuit(command)) {
    System.out.println("Вы вышли из игры");
    return;
  } 
  start();
}

private static void start() {
  //игровая логика раунда
}

private static String inputCommand() {
  while(true) {
    String command = scanner.nextLine().toUpperCase();
    if(isStart(command) || isQuit(command)) {
      return command;
    }
    System.out.println("Вы вводите неверное значение!!!");
  }
}
```
