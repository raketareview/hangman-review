Введенные буквы должны размещаться по алфавиту(й, к, ц) либо в порядке ввода(й, ц, к).

**3. Текст в exception всегда должен быть на английском языке**
```java
throw new RuntimeException("нет слов для игры");
```
Исключение это не просто телеграмма, которая летит сквозь слои. 
У exception особое назначение- если исключение вылетит и не будет перехвачено внутри программы, то аварийно прекратит выполнение программы.
Тогда на экране будет распечатано сообщение эксепшена, и это сообщение должно быть понятно сисадмину в любой точке планеты. 
А значит, сообщение должно быть на английском. 
Интерпритация исключения и перевод его на локальный язык должно происходить там, где это соответствует архитектуре программы или не происходить вовсе, если исключение не планируется перехватывать.

**4. Если в блоке if есть return(break, continue, throw, exit и т.д.), то else не пишется.**
 
В этом случае неважно, будет else или нет, так как программа будет работать одинаково, а код без `else` будет выглядеть читабельней
```java
if (isWon()) {
  return GameState.WON;
} else {
  return GameState.LOST;
}

//ПРАВИЛЬНО:
if (isWon()) {
  return GameState.WON;
} 
return GameState.LOST;
```

**5. class TextValidator**

+ 👍 Валидатор для текстов это хорошо. Благодаря этому можно использовать разные языки в программе.

- Нарушение SRP.

Валидатор должен только валидировать, то есть проверять данные на соответствие каким-то требованиям.  
А этот метод не имеет к валидации никакого отношения и не должен быть в этом классе:
```java
public String toCanonicalForm(String text) {
  return text.trim().toLowerCase();
}
```

- Букву нужно валидировать как char, а не как String
```java
public boolean isSingleLetterValid(String text)

//ПРАВИЛЬНО:
public boolean isValidLetter(char symbol)
```

- Процедурный стиль.

Я бы сказал, что класс сделан в процедурном стиле.  
В нем есть два метода, которые делают похожее, но разное- валидация буквы и валидация слова.   
Каждый метод обладает своими правилами и особенностями.

Раз уж хочешь сделать хангмана по-серьезному с валидаторами, то я бы предложил это сделать в ООП стиле:  
Создать папку `validator`.  
Создать базовый абстрактный класс валидации, типа такого
```java
public abstract class Validator<T> {
  //что-то хранит и принимает в конструктор
  
  public abstract boolean isValid(T value);
}
```

От него сделать два наследника:
```java
public class WordValidator extends Validator<String>{
  @Override
  public boolean isValid(String value) {...}
}

public class LetterValidator extends Validator<Character>{
  @Override
  public boolean isValid(Character value) {...}
}
```

И уже в каждом из этих наследников разместить логику, по которой валидируется слово(количество букв в слове и т.д.) и вводимая юзером буква.  
Да, эта логика будет простой, а классы валидации- маленькими. Но с точки зрения ООП архитектуры так будет чище и лучше.

Валидатор букв должен валидировать букву как `Character`, а не как `String`.  
То есть клиент, прежде чем отдать букву на валидацию, должен сам увидеть, что строка состоит из одного символа.  
Потом взять этот символ как `char` и передать на валидацию.

Потому что буква это один символ и валидация буквы, как концепция, это валидация одного символа.  

**6. class WordRepository**

- Реакция на ошибки

Если по указанному адресу не будет найден файл со словами, то программа аварийно вылетит и напечатает непонятные для юзера сообщения в консоль
```java
Exception in thread "main" java.lang.RuntimeException: файл со словами не найден: /words.txt
	at org.example.word.WordRepository.readLinesFromFile(WordRepository.java:37)
    at org.example.word.WordRepository.<init>(WordRepository.java:16)
```
Здесь должна быть другая реакция на отсутствие файла со словами.  
Нужно сказать юзеру, что файл со словами по указанному пути открыть не удалось и поэтому работа программы будет завершена. 

Причем, путь нужно указать не относительный, как у тебя(`/words.txt`).  
А абсолютный, чтобы юзер знал, где конкретно ему нужно искать файл: `c:/..../words.txt`.

После этого корректно завершить работу программы.

- Приоритет- читаемость кода. 

Не стремись несколько иструкций всегда писать в одну строку.  
Иногда код читается легче, когда в каждая инструкция находится в отдельной строке
```java
public WordRepository(String fileName, TextValidator textValidator) {
  this.words = filterWords(readLinesFromFile(fileName), textValidator);
}

//ЛУЧШЕ:
public WordRepository(String fileName, TextValidator textValidator) {
  List<String> allWords = readLinesFromFile(fileName);  
  this.words = filterWords(allWords, textValidator);
}
```

- Побочный эффект.

Метод неявно вносит изменение в поле класса
```java
private List<String> words = new ArrayList<>();

public WordRepository(String fileName, TextValidator textValidator) {
  this.words = filterWords(readLinesFromFile(fileName), textValidator);  
}
 
private List<String> filterWords(List<String> lines, TextValidator textValidator) {
  words = lines.stream()  <-- НЕЯВНО МЕНЯЕТ ПОЛЕ КЛАССА this.words
  .map(word -> textValidator.toCanonicalForm(word))
  //...
  return words;
}
```

Или метод `filterWords(...)` должен заполнить `this.words` данными и при этом ничего не возвращать, то есть быть `void`.  
Или должен внутри себя создать список отфильтрованных слов и вернуть его. Например, так:
```java
private List<String> words;

public WordRepository(String fileName, TextValidator textValidator) {
  this.words = filterWords(readLinesFromFile(fileName), textValidator);
}

private List<String> filterWords(List<String> lines, TextValidator textValidator) {
  List<String> filteredWords = new ArrayList<>();
  filteredWords = lines.stream()  
  .map(word -> textValidator.toCanonicalForm(word))
  //...
  return filteredWords;
}
```

*Фаулер "Рефакторинг", гл.6, "Извлечение метода"* 

+ 👍 В целом, в классе реализована правильная идея использования валидатора слов, который инжектится в конструктор.  
Благодаря этому, `WordRepository` становится универсальным и может использоваться для загрузки слов на разных языках.

**7. enum HangmanState**, картинки хангмана

- Не переопределяй `toString()` у енамов.

Хотя прямого запрета на переименование `toString()` у енамов я не встречал, но этот метод в классах принято использовать только для отладки.  
В этом енаме переопределенный `toString()` возвращает картинку, а значит предназначен не для отладки, а для использования в представлении.  
Я бы советовал у енамов не переопределять `toString()`, он в енамах и так по умолчанию хорош для задач отладки.

Про особенности использования метода `toString()` почитай тут:  
*Блох, "Java. Эффективное программирование", изд.3, гл.3.3*.

В этом енаме для возврата картинки вместо `toString()` сделай обычный геттер
```java
@Override
public String toString() {
  return state;
}

//ПРАВИЛЬНО:
public String getState() {
  return state;
}
```

- Неправильное использование енама.

Этот енам в проекте не используется как перечисление констант.  
Клиенты используют этот енам только через вызов метода 
```java
public static HangmanState fromMistakes(int mistakes) {
  if (mistakes >= values().length) {
    return values()[values().length - 1];
  }
  return values()[mistakes];
}
```
Цель существования этого енама- не хранить константы, а выдавать картинки по номеру через метод со сложной логикой.

Енамы нужно использовать как "константы на стероидах", в них не должно быть сложной логики.  
[Документация](https://docs.oracle.com/javase/8/docs/technotes/guides/language/enums.html) на enum сообщает:
```java
Итак, когда следует использовать перечисления (enums)? Всякий раз, когда вам нужен фиксированный набор констант.
```
От этого енама нам нужен не фиксированный набор констант, а получение картинки по номеру ошибки.

Вместо енама здесь корректнее использовать обычный класс.  
Это будет и правильнее с точки зрения архитектуры, и проще для пользования.  
Например:
```java
public class PictureStorage {

  private static final String[] PICTURES = {
      """
       +----
       |   
       |    
       |    
       |    
     =======
     """
      ,
      """
       +---+
       |   |
       |   0   
       |
       |
     =======
     """
   // more pics
  };

  public String get(int pictureNumber) {  
    return PICTURES[pictureNumber]; 
  }
}
```

Но будет ещё лучше добавить в этот класс метод распечатки картинки, тогда получится класс-рендерер:
```java
public class HangmanRenderer {

  private static final String[] PICTURES = {
      """
       +----

       |

     """
      ,

       |   0

  };

  public void render(int pictureNumber) {  
   String picture = PICTURES[pictureNumber]; 
    System.out.println(picture);
  }

```

- Неправильная реакция на ошибки.

Есть три основных реакции на ошибки: игнорирование, исправление, падение.  
Здесь выбрано исправление- при получении некорректного номера картинки, метод подгоняет номер под последний корректный
```java
public static HangmanState fromMistakes(int mistakes) {
  if (mistakes >= values().length) {
    return values()[values().length - 1];
  }
  return values()[mistakes];
}
```

Правильная реакция должна быть другой, нужно сделать падение:
```java
public static HangmanState fromMistakes(int mistakes) {
  return values()[mistakes];  <-- БРОСИТ ИСКЛЮЧЕНИЕ при некорректном значении mistakes
}
```
Потому что запрос картинки с неправильным номером означает, что в алгоритме игры есть баг.  
Правильно работающий алгоритм не будет пытаться получить картинку с несуществующим номером.

Этот баг не нужно маскировать.  
Это приведет только к тому, что игра будет происходить некорректно- картинка не будет соответствовать количеству ошибок.  
Баг нужно максимально быстро выявить и устранить.

## АРХИТЕКТУРА

Архитектура в целом соответствует ООП.  
Но выявленные и сделанные в виде классов сущности довольно очевидны- Репозиторий слов, Игра, Хранилище картинок и т.д.  

Хотелось бы видеть более продвинутое понимание ООП.  
Нужно понять, что отгадываемое слово, его маска и методы работы с ними(открыть букву и др.) это отдельная сущность.  
И эту сущность тоже нужно сделать классом.
