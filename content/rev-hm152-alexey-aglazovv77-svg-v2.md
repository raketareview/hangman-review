## ЗАМЕЧАНИЯ

**1. Нейминг**

- Название метода может соответствовать названию переменной только в определенных случаях.  
И тут не тот случай:
```java
private static List<Character> errorLeters;

private static List<Character> errorLeters(char letter) {
  errorLeters.add(letter);
  return errorLeters;
}

//ПРАВИЛЬНО:
private static List<Character> errorLeters;

private static List<Character> addErrorLeter(char letter) {...}
```
Пока не знаешь тонкостей, когда можно называть методы так же как переменные, просто не называй их одинаково.  
Лучше пусть методы и поля будут всегда называться по-разному, от этого хуже точно не будет.

- Обращай внимание на подсказки Idea. Если название переменной подчеркивается зеленым- значит в названии есть орфографическая ошибка
```java
List<Character> errorLeters;

//ПРАИВЛЬНО:
List<Character> errorLetters;
```

**2. class ReadWords**

- Соблюдай требования утилитных классов. 

Это утилитный класс, он содержит только статические методы.  
Утилитные классы должны быть `final` и иметь приватный конструктор.  
Не должно быть возможности унаследоваться от утилиты или сделать ее экземпляр.  
*Блох, "Java. Эффективное программирование", изд.3, гл.2.4*

- Название класса должно быть существительным. Это название- глагол
```java
class ReadWords

//ПРАВИЛЬНО:
class WordReader  //или WordStorage, FileReader, Dictionary etc.
```

- Метод класса не должен называться так же, как сам класс. 

Первая причина: название класса должно быть существительным, а методов- глаголами в повелительном наклонении.  
Вторая причина: называться так же, как класс, может исключительно один вид методов- конструктор этого класса
```java
public class ReadWords {
  public static List<String> readWords() {...}
  //...
}

//ПРАВИЛЬНО:
public class WordReader {
  public static List<String> read() {...}
  //...
}
```

- Это абсолютный адрес, поэтому класс будет корректно работать только на одном компьютере в мире- на твоем.  
На моем компьютере этот класс работать не будет потому что у меня нет такого пути `"C:\\Users\\aglaz\\..."`
```java
private final static String FILE_PATH = "C:\\Users\\aglaz\\ProjectsIdea\\HangMan\\src\\main\\resources\\words.txt";
```

- Параметризация.

Даже в процедурном стиле, при прочих равных нужно делать методы универсальными.  
Универсальный метод лучше неуниверсального метода своей универсальностью
```java
public static List<String> readWords() {...}

//ПРАВИЛЬНО:
public static List<String> read(String filepath, String regex) {...}
```
Здесь, если метод будет принимать эти параметры, можно будет бесплатно использовать его для чтения разных словарей(напр. инженерный, географический) и на разных языках(английский, русский).  

- Текст в exception всегда должен быть на английском языке
```java
throw new IllegalArgumentException("Файл не найден, работа программы будет прекращена");

//ПРАВИЛЬНО:
String path = .... //полный путь к файлу
throw new IllegalArgumentException("File not found: " + path);
```
Исключение это не просто телеграмма, которая летит сквозь слои. 
У exception особое назначение- если исключение вылетит и не будет перехвачено внутри программы, то аварийно прекратит выполнение программы.
Тогда на экране будет распечатано сообщение эксепшена, и это сообщение должно быть понятно сисадмину в любой точке планеты. 
А значит, сообщение должно быть на английском. 
Интерпритация исключения и перевод его на локальный язык должно происходить там, где это соответствует архитектуре программы или не происходить вовсе, если исключение не планируется перехватывать.

Информация в исключении должна быть информативной.  
В данном случае нужно указать полный путь к файлу, который не удалось прочесть.

- Компилируй регулярные выражения. 

Если регулярное выражение используется многократно, как здесь, то нужно его откомпилировать.  
Компиляция регулярного выражения в объект Pattern с помощью Pattern.compile() позволяет повысить производительность при многократном использовании
```java
while (...) {
  if (!word.matches("[А-Яа-яЁё]+")) {
}

//ПРАВИЛЬНО:
private static final String REGEX = "[А-Яа-яЁё]+";
private static final Pattern PATTERN = Pattern.compile(REGEX);

while (...) {
  Matcher matcher = PATTERN.matcher(word);
  if (!matcher.matches("[А-Яа-яЁё]+")) {
}
```
Блох пишет:
```java
Хотя String.matches — простейший способ проверки, соответствует ли строка регулярному выражению, 
он не подходит для многократного использования в ситуациях, критичных в смысле производительности. 
```
Здесь неиспользование компиляции сильно подрывает производительность метода потому что проверка на регулярное выражение проводится для каждого считанного из файла слова, 
а их могут быть тысячи и десятки тысяч.  
*Блох "Java. Эффективное программирование", изд.3, гл.2.6*

```java
private static String openLetter(String word, String mask, char letter) {

return String.valueOf(maskArray);
```

- Побочный эффект. Метод неявно меняет состояние поля класса
```java
private static String getMaskWord(String word) {
  String s = String.valueOf(MASK_SYMBOL);
  return mask = s.repeat(word.length()); <-- УСТАНАВЛИВАЕТ ПОЛЕ КЛАССА И ОДНОВРЕМЕННО ВОЗВРАЩАЕТ ЕГО
```

- Метод выполняет две операции, совмещение команды и запроса
```java
private static String openLetter(String word, String mask, char letter)

//ПРАВИЛЬНО:
private static void openLetter(String word, String mask, char letter)
```
Команда: открыть букву.  
Запрос: сказать, есть ли эта буква в слове.

Метод должен или выполнять команду, или отвечать на запрос.  
У тебя уже есть специальный метод, который проверяет наличие буквы в слове:
```java
boolean isWordLetter(char letter)
```

*Мартин, "Чистый код", гл.3, "Разделение команд и запросов"* 
```java
"Либо функция изменяет состояние объекта, либо возвращает информацию об этом объекте. 
Совмещение двух операций часто создает путаницу." - Мартин.
```
Принцип разделения команд и запросов это общее правило. 
Если ты считаешь, что в данном конкретном случае совмещение команды и запроса делает код проще и понятнее, то совмещай.  
Нужно просто знать, что такое правило существует.

- Побочные эффекты носят системный характер. Избавься от такого подхода:
```java
private static List<Character> errorLeters(char letter) {
  errorLeters.add(letter);
  return errorLeters;
}
```

- При прочих равных, бери данные из окружения класса. 

Если одни и те же данные используются в разных методах класса, то эти данные нужно не перекидывать туда-сюда между методами, а сделать их полями класса
```java
private static String word = "";
private static String mask = "";

private static String openLetter(String word, String mask, char letter) {
  char[] maskArray = mask.toCharArray();
  for (int i = 0; i < word.length(); i++) {
    if (letter == (word.charAt(i))) {
      maskArray[i] = letter;
    }
  }
  mask = String.valueOf(maskArray);
  return mask;
}

//ЗДЕСЬ ЛУЧШЕ:
private static void openLetter(char letter) {
  char[] maskArray = mask.toCharArray();
  for (int i = 0; i < word.length(); i++) {
    if (letter == word.charAt(i)) {
      maskArray[i] = letter;
    }
  }
  mask = String.valueOf(maskArray);
}
```
Да, вопрос о том, в каких случаях и когда поля должны браться из окружения, а когда передаваться между методами, дискуссионен и не так прост.
Об этом нужно думать отдельно в каждом конкретном случае.

Но в данном случае эти поля уже являются полями класса, так сказать "глобальными переменными" в концепции процедурного программироваия.  
А поэтому к ним можно обращаться напрямую, то есть "брать их их из окружения".
