**2 enum GallowsStage**

Здесь хранятся картинки хангмана.

Перечисления должны использоваться как константы- потому что они и были придуманы, как разновидность констант.  
Не нужно делать из енамов недоклассы.

Сейчас этот енам не используется как енам- к его константам другие классы не обращаются напрямую, а юзают метод `String getArtByErrors(int errors)`

- Используй строки так, чтобы их можно было нормально прочитать.

Код должен читаться легко и приятно.  
Не заставляй читателя самому делать сложных преобразований в своей голове.  
Например разворачивать 1D в 2D

- Неправильная реакция на ошибки. 

Есть три основных реакции на ошибки: игнорирование, исправление, падение.  
Здесь выбрано исправление- при получении некорректного номера картинки, метод подгоняет номер под последний корректный
```java
public static String getArtByErrors(int errors) {
  for (GallowsStage stage : values()) {
    if (stage.errors == errors) {
      return stage.asciiArt;
    }
  }

  return STAGE_6.asciiArt;
}
```

Правильная реакция должна быть другой, нужно сделать падение:
```java
public static String getArtByErrors(int errors) {
  for (GallowsStage stage : values()) {...}

  throw new ПодходящееСлучаюИсключение(...);
}
```
Потому что запрос картинки с неправильным номером означает, что в алгоритме игры есть баг.  
Правильно работающий алгоритм не будет пытаться получить картинку с несуществующим номером.

Этот баг не нужно маскировать.  
Это приведет только к тому, что игра будет происходить некорректно- картинка не будет соответствовать количеству ошибок.  
Баг нужно максимально быстро выявить и устранить.

### Пакет service

Здесь должны быть сервисы.  
То есть классы, которые содержат бизнес-логику, не хранят данные и не завися от представления.  
Единственный класс в этом пакете- не сервис.   

**4. class Dictionary**

- Это не сервис- этот класс хранит данные и зависит от представления.

- Путь к файлу класс должен получать в конструктор. 

Иначе класс становится не универсальным.  
Допустим, в программе будет несколько разных текстовых файлов- на разных языках или по разным темам.  
Тогда вместо нескольких специализированных классов Словаря, где имя файла словаря будет жестко прописано в каждом из классов, 
можно будет использовать один универсальный класс Словарь.

- Нарушение SRP, зависимость от представления

Модели и сервисы(которые по сути являются просто разновидностью моделей) не должны ничего печатать в консоль или куда-то еще. 
То есть, не должны зависеть от представления
```java
System.out.println("[Ошибка] Ошибка при чтении файла: " + e.getMessage());
```
Потому что бизнес-логика сервисов должна одинаково работать в любом окружении: консоль, веб, Андроид. 
И под любым другим визуалом, вплоть до аркадного автомата.

Сервисы и модели не должны ничего печатать самостоятельно.

- Использование исключений.

Ну ок, если прочитать словарь не получится, то юзеру напечатают сообщение
```java
try (...) {
  //... 
} catch (IOException e) {
  System.out.println("[Ошибка] Ошибка при чтении файла: " + e.getMessage());
}
```

А дальше что? Программа все равно вылетит с исключением
```java
=========================================
             Игра началась!              
=========================================
[Ошибка] Ошибка при чтении файла: src\main\resources\fuckingwords.txt (Не удается найти указанный файл)
Exception in thread "main" java.lang.IllegalArgumentException: bound must be positive
```

Здесь должна быть другая реакция на отсутствие файла со словами.  
Нужно сказать юзеру, что файл со словами по указанному пути открыть не удалось и поэтому работа программы будет завершена. 

Причем, путь нужно указать не относительный, как у тебя: `src\main\resources\fuckingwords.txt`.  
А абсолютный, чтобы юзер знал, где конкретно ему нужно искать файл: `c:/..../fuckingwords.txt`.

После этого корректно завершить работу программы.

- Нарушение SRP. Метод не должен завершать работу программы через exit()
```java
private int[] getWorldParameters() {
  //...  
  System.exit(0);
}  
```
Каждый метод и класс имеют право завершать только свою работу. Например, через return.  
Потому что методы и классы не должны знать логику работу более высоких слоев программы, 
у которых могут быть свои планы на тему того, когда и почему нужно завершать работу программы.

- Нарушение SRP.

Если это класс меню, то его задача должна состоять только в том, чтобы юзер выбрал пункт и меню и выполнить команду этого пункта: выйти из программы или начать игру.  
Этот класс сейчас не только запускает игру, НО И ВЫПОЛНЯЕТ САМ ПРОЦЕСС ИГРЫ.
