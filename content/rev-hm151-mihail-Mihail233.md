**1. Нейминг**

- Константы должны быть static final
```java
private int MAX_NUMBER_OF_ATTEMPTS = 6;

//ПРАВИЛЬНО:
private static final int MAX_NUMBER_OF_ATTEMPTS = 6;
```

- Название должно объяснять, что делает метод.

"Проверить букву"- ничего не объясняет.  
Из названия непонятно, как именно проверяется буква- это проверка на то, что она первая буква в алфавите? Большая? Гласная? Испанского языка?  
На самом деле метод проверяет, есть ли эта буква в слове
```java
class RandomWord {
  boolean checkLetter(String letter) {...}
  //...
}

//ПРАВИЛЬНО:
class RandomWord {
  boolean hasLetter(String letter) {...}
  //...
}
```

- Название обманывает. Метод не только проверяет ответ(что бы это ни значило), но еще выставляет какие-то флаги
```java
boolean checkAnswer(String answer)
```

- Буква не может быть String. Потому что буква это один символ, а строка- ноль, один и более символов.  
Эта переменная должна либо называться по-другому, либо не быть String
```java
String letter

//ПРАВИЛЬНО:
char letter
```

- Если в проекте есть класс `RandomWord` или `PictureHangman`, то все переменные с именем, включающим это название, должны быть экземплярами этого класса.
Если разные концепции называются одним и тем же именем, это приводит к путанице
```java
public class RandomWord {
  private String randomWord;
  //...
}

public class PictureHangman {
  private final List<String> pictureHangman;
  //...
}

//ПРАВИЛЬНО:
public class RandomWord {
  private String text;
  //...
}

public class PictureHangman {
  private final List<String> lines;
  //...
}
```

- Венгерская ноттация.

В названии переменных не пиши тип данных, к которым они относится. И вообще не употребляй венгерскую ноттацию.  
Название переменной должно отвечать на вопрос что хранит переменная, а не как хранит.  
Массивы и коллекции должны называться во множественном числе
```java
List<String> listRandomWord

//ПРАВИЛЬНО:
List<String> words
```

- "Write" это что-то из Паскаля. Здесь лучше "print" или "show"
```java
void writeHangman(Game game)
```

- Одной концепции давай одно название. Определись, как ты хочешь называть методы распечатки- "print", "write" или "show"
```java
void printResultTheGame()
void writeHangman(Game game)
void showResultOfOneTurn(boolean isFind)

//ПРАВИЛЬНО:
void printResultTheGame()
void printHangman(Game game)
void printResultOfOneTurn(boolean isFind)
```

- Сеттер должен принимать устанавливаемое значение в аргументы. Если не принимает- это не сеттер
```java
public void setRoundNumber() {
  roundNumber++;
}

//ПРАВИЛЬНО:
public void incRoundNumber() {
  roundNumber++;
}
```

- Исключения в блоке `catch` принято называть "e" или "ex"
```java
catch (IOException err)

//ПРАВИЛЬНО:
catch (IOException e)
```

**2. Форматирование строк**

Если нужно печатать или создавать строку с более, чем одним подстановочным значением или значение вставляется внутрь сообщения, используй форматирование- тогда сразу будет виден весь шаблон
```java
System.out.println("Вам загадали слово из " + randomWord.length() + " букв");

//ПРАВИЛЬНО:
System.out.printf("Вам загадали слово из %d букв  %n", randomWord.length());
```

**4. class RandomWord**

- Позиционирование.

Этот класс не представляет из себя какую-то сущность с четкими границами.  
Он делает разное, поэтому с точки зрения ООП, это не объект.  
Это контейнер для процедур и переменных.

Назначение класса:  
Прочитать случайное слово из файла и хранить методы для работы с этим словом.

Чтобы здесь появилось ООП, класс нужно раздедить на два.  
Один класс должен читать файл со словами и возвращать случайное слово.  

Второй класс должен представлять из себя сущность "Секретное слово".  
То есть, такая строка, которая имеет скрытые буквы и открытые буквы(маска), и методы для работы с этими полями.

- Путь к файлу класс должен получать в конструктор. 

Иначе класс становится неуниверсальным.
Допустим, в программе будет несколько разных текстовых файлов- на разных языках или по разным темам. 
Тогда вместо нескольких специализированных классов, где имя файла словаря будет жестко прописано в каждом из классов, 
можно будет использовать один универсальный класс.

- Проверямые исключения перехватывай в точке их возникновения и заменяй на непроверяемые.

Проброс исключений в сигнатуре методов захламляет код.  
Поэтому заменяй проверяемые исключения на непроверяемые.  
Изучи различия проверяемых и непроверяемых исключений 
```java
public String getRandomWordFromFile() throws IOException {
  List<String> listRandomWord = Files.readAllLines(Paths.get("./src/resources/randomWord.txt"));
 //...
}

//ПРАВИЛЬНО:
public String getRandomWordFromFile(){
  try {
    listRandomWord = Files.readAllLines(Paths.get("./src/resources/randomWord.txt"));
  } catch (IOException e) {
    throw new НепроверяемоеИсключение(e);
  }
  //...
}
```
*"ЧК", гл.7, "...проверяемые исключения"*

- Этот метод делает две операции сразу.

Методы должны делать только одну операцию.
```java
public String getRandomWordFromFile(){
  //...
  randomWord = chooseRandomWord(listRandomWord);  <-- ОПЕРАЦИЯ #1, найти и позднее вернуть случайное слово
  System.out.println("Вам загадали слово из " + randomWord.length() + " букв");  <-- ОПЕРАЦИЯ #2, напечатать в консоль информацию 
  return randomWord;  
}
```
*Мартин, "Чистый код", гл.3, "Правило одной операции"*

- Нарушение SRP, чужая ответственность, зависимость модели от представления.

Модель(а это модель) не должна ничего печатать в консоль
```java
public String getRandomWordFromFile(){
  //...
  System.out.println("Вам загадали слово из " + randomWord.length() + " букв");  
}
```
Иначе модель перестает быть универсальной и становится заточенной под конкретную среду и конкретное представление себя в этой среде- в данном случае, консоль.
В других средах нужно будет менять код в этом классе, чтобы вывод осуществлялся по правилам этой среды, то есть, это лишняя причина для изменения класса.
В других средах(напр. Андроид) эту модель нельзя будет использовать- она там просто не скомпилируется. 
Другое представление для модели(напр. если одну и ту же модель нужно в программе показвать по-разному) нельзя будет сделать, или придется делать через костыль.

- Неправильный toString(). 

toString() должен быть стандартным(как делает IDE по Alt+Ins) и использоваться только для отладки.  
toString() должен содержать значение всех значимых полей класса.  
toString() не должен использоваться для представления, за исключением предельно простых классов, вроде PhoneNumber
```java
@Override
public String toString() {
  return String.join(", ", getOpenLetters());
}

//ПРАВИЛЬНО:
@Override
public String toString() {
  return "RandomWord{" +
    "hiddenForm=" + hiddenForm +
    //...
   '}';
}
```
*Блох, "Java. Эффективное программирование", изд.3, гл.3.3*

- Нарушение SRP.

- Нарушение Low Coupling 

Этот класс не должен знать про существование класса Game.  
Он должен печатать картинку виселицы по ее номеру

```java
public void writeHangman(int pictureNumber) {...}
```

- Циклическая зависимость.

Game зависит от PictureHangman, а PictureHangman зависит от Game
```java
public class Game {

  //...

public class PictureHangman {
  public void writeHangman(Game game)  <-- Зависимость от Game
  //...

```
По возможности избегай циклических зависимостей между классами.  
*Фаулер "Рефакторинг", гл.8, "Замена двунаправленной связи однонаправленной"*

- Спагетти код, побочный эффект.

Но плохо не только то, что между этими классами циклическая связь.  
Еще хуже то, что класс картинки внезапно меняет состояние класса Game

Задача этого метода должна состоять просто в том, чтобы напечатать картинку.  
Все остальные действия в этом методе будут являться побочным эффектом.

- Нечитабельный код.

Названия методов не соответвуют тому, что они делают, поэтому код не читается.  
Например, этот метод не выполняет никакую проверку- потому что он void
```java
public void checkStateGame() throws IOException {
  if (!stateGame) {
    System.exit(0);
  } else {
    pictureHangman = new PictureHangman();
    randomWord = new RandomWord();
  }
}
```
Метод проверки должен возвращать результат проверки в виде boolean.

Не нужно в любой непонятной ситуации называть методы словом "check": `checkStateGame()`, `checkAnswer()`.  
Могу предположить, что ты так называешь методы тогда, когда не можешь сам для себя сформулировать, что они делают.  
Если это так, то скорее всего, методы делают что настолько разное, что это нельзя обобщить в названии.

Если метод реально что-то проверяет, то по конвенции кода его в большинстве случаев следует называть "is" или "has".  

Если метод что-то проверяет и на основе этой проверки внутри себя что-то делает(например, инициализирует данные раунда), 
то это не "check". Такой метод нужно назвать так, чтобы название описывало то, что реально делает метод.

Если названия методов не описывают то, что на самом деле делают методы, то алгоритм программы становится нечитаемым.

**7. Main.java**

- Класс Main отсутствует.  

Есть просто файл "Main.java", в котором нет класса Main, а есть только метод main:
```java
import com.hangman.Game;

void main() {
  //...
}
```
В java нет "глобальных процедур", как в других языках.  
Все методы являются поведением какого-то класса.  
Поэтому должны находиться в классах, даже если ловкими фокусами можно сделать иначе:
```java
public class Main {
  public static void main(String[] args) {
    //...
  }
}
```

- Отделяй создание от использования.

Метод main управляет процессом игры
```java
void main() {
  try {
    while (true) {
      Game game = new Game();
      game.startGame();
      game.guessWord();
      game.printResultTheGame();
    }
  } catch (IOException err) {...}
}
```
Метод main это точка входа, он должен только конфигурировать и запускать игру, примерно так:
```java
  public static void main(String[] args) {
    try {
      Game game = new Game();
      game.start();
    } catch (IOException e) {...}
  }
```
Если игра идет через диалог "играть или выйти", то этот диалог тоже должен вестись в main.  
*Мартин "ЧК", "Отделение конструирования системы от ее использования"*

## АРХИТЕКТУРА

Несмотря на то, что в проекте несколько классов, он написан в процедурном стиле.  
Программа в ООП стиле должна быть декомпозирована по правилам ООП и использовать ООП подход для реализации типичных задач.  
Здесь не видно ООП декомпозиции.

Алгоритм программы тяжело читается из-за плохого нейминга, побочных эффектов и спагетти кода.

Методы нужно переделать так, чтобы:
+ Название соответствовало тому, что делает метод
+ Каждый метод делал одну операцию и не совмещал команду и запрос
+ Отсутствовали побочные эффекты
