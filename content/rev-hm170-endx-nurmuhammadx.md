## ЗАМЕЧАНИЯ

**1. Нейминг**

- Если в проекте есть класс `Word`, то все переменные с именем, включающим это название, должны быть экземплярами этого класса.  
Если разные концепции называются одним и тем же именем, это приводит к путанице
```java
public class Word {
  private final String targetWord;
  //...
}

//ПРАВИЛЬНО:
public class Word {
  private final String text;
  //...
}
```

- Не называй метод "run" если он не переопределяет `Runnable.run()`- название этого метода ассоциируется с этим стандартным интерфейсом и применением потоков.  
Здесь этот метод не имеет отношения к `Runnable.run()`
```java
public class ConsoleView {
  public void run() {...}
  //...
}
```

- Константы должны быть static
```java
private final int STATES = 5;

//ПРАВИЛЬНО:
private static final int STATES = 5;
```

*Oracle Java code conventions, part."Naming conventions"*

**2. Нарушение конвенции кода**

- Скобочки.

В любой ситуации выделяй тело блока скобочками, даже если тело состоит из одной строки.  
Исключение- метод equals(), там можно после if не выделять блоки скобочками
```java
if (words.isEmpty()) throw new IllegalStateException("Dictionary is empty");

//ПРАВИЛЬНО:
if (words.isEmpty()) {
  throw new IllegalStateException("Dictionary is empty");
}
```
*"Oracle Java code conventions"*

**4. Нарушение DRY**

- Магические буквы, числа, слова. Вводи константы 
```java
case 1 -> startGame();
case 0 -> {...}
System.out.println("Incorrect choice, enter 1 or 0!");

//ПРАВИЛЬНО:
private static final int QUIT = 0;
private static final int START = 1;

case START -> startGame();
case QUIT -> {...}
System.out.printf("Incorrect choice, enter %d or %d!  %n", START, QUIT);
```

- Если к одной и той же магической штуке обращаются два разных класса, то делай ее константой в одном из этих двух классов.  
Либо перенеси ее в третий класс и из первых двух классов обращайся к этим константам- эти данные должны быть синхронизированы между собой
```java
public enum MenuOption {
  NEW_GAME(1, "New Game"),
  EXIT_GAME(0, "Exit");
  //...
}

public class ConsoleView {
  //...

  public void run() {
    case 1 -> startGame(); 
    case 0 -> {...}
    default -> System.out.println("Incorrect choice, enter 1 or 0!");
    //...
  }
}
```

*refactoring.guru "Замена магического числа символьной константой"*

**5. Форматирование строк**

- Мысленные преобразования.

Не вставляй перевод строки в середину строк при распечатке.  
Иначе при чтении кода приходится мысленно "разворачивать" одну строку в несколько.  
Если нужно распечатать несколько строк, то печатай их или многострочными строками(""") или отдельными инструкциями печати.  
Тогда сразу будет видно, как текст будет выглядеть на печати:
```java
System.out.println("\n=== Game ===");

System.out.println("\nYour lives: " + gameService.getLives());

//ПРАВИЛЬНО:
System.out.println();
System.out.println("=== Game ===");

System.out.println();
System.out.println("Your lives: " + gameService.getLives());
```

**6. Соблюдай требования утилитных классов**

Утилитные и константные классы должны быть `final` и иметь приватный конструктор
```java
public class GallowsGraphics {
  private GallowsGraphics() {...}
  //...
}

//ПРАВИЛЬНО:
public final class GallowsGraphics {
  private GallowsGraphics() {...}
  //...
}
```
Не должно быть возможности унаследоваться от утилиты или сделать ее экземпляр.

### ВАЛИДАТОРЫ

Классы валидаторов должны содержать бизнес логику, которая не зависит от представления.  
Поэтому они относятся к моделям(MVC) или к сервисам(MVCS).  
Здесь пакет с валидаторами: validation.

**9. class InputValidator**

- Нейминг.

Валидатор должен что-то валидировать.  
То есть проверять данные на соответствие каким-то требованиям. 

Этот класс не является валидатором, потому что он не валидирует.  
Его назначение в другом- он осуществляет ввод данных от юзера
```java
public class InputValidator {
  private InputValidator() {...}

  public static int getValidInt(Scanner scanner) {
    while (!scanner.hasNextInt()) {
      System.out.print("Please enter an integer: ");
      scanner.next();
    }
    return scanner.nextInt();
  }
}
```
То есть, это инпутер.

- Принадлежность к слою.

То, что это не валидатор, а инпутер, влечет за собой конкретные последствия.

Валидаторы это модели(MVC) или сервисы(MVCS) и поэтому они не должны зависеть от представления.  
А класс `InputValidator` зависит- он печатает напрямую в консоль
```java
System.out.print("Please enter an integer: ");
```

Инпутер это вью.

### ПРЕДСТАВЛЕНИЕ (VIEW)

В архитектуре MVC вью должны распечатывать данные, в том числе модели.  
Также вью должны принимать ввод от юзера.  
Вью не должны ничего знать про контроллеры.

Здесь пакет с вью: view.

**10. class ConsoleView**

- Это не вью.

По своему функционалу, это не вью, это контроллер.  
Причем, контроллер толстый.

Классы-вью должны быть достаточно тупыми классами- их задача просто распечатывать информацию и принимать данные от юзера.  
Класс `ConsoleView` не только распечатывает инфу на экран, но и содержит сложную логику управления игрой.

Класс нужно разделить на два: контроллер и вью.  

Вью должен содержать методы для распечатки инфы на экран.  
И вся распечатка информации на экран должна идти через этот вью-класс.

Контроллер должен использовать вью и модели с сервисами для работы всей программы.  
При этом контроллер всю распечатку должен делать только через классы-вью.

- Вью не должно управлять сервисом, это противоречит задачам слоя вью
```java
public class ConsoleView {
  //...

  private void startGame() {
    //...
    gameService.processLetter(letter);
  }
}
```

- Нарушение правила одной операции. 

Этот метод выполняет две операции:  
Определяет состояние игры и распечатывает состояние игры
```java
private void printResult() {
  if (gameService.getLives() == 0) {
    //печатает сообщение при проигрыше
  } else {
    //печатает сообщение при выигрыше
  }
}
```

Метод нужно разделить на несколько, каждый из которых будет делать что-то одно.  
Например:
```java
private boolean isWin() {...}
private boolean isLose() {...}

private boolean isGameOver() {
  return isWin() || isLose();
}

private void printWinMessage() {...}
```
*Мартин, "Чистый код", гл.3, "Правило одной операции"*

- Диалог "Играть или выйти".

Диалог "Играть или выйти" должен происходить не в этом классе, а на уровень выше.  
Иначе, если юзер сразу выберет "выйти", значит текущий экземпляр этого класса был создан зря.

- Вводи поясняющие переменные, это не читается:
```java
System.out.println(GallowsGraphics.STAGES.get(STATES - gameService.getLives()));

//ПРАВИЛЬНО:
int pictureIndex = STATES - gameService.getLives();
String picture = GallowsGraphics.STAGES.get(pictureIndex);
System.out.println(picture);
```

**11. class GallowsGraphics**

+ 👍 Да, это вью. Здесь просто хранится публичная константа с массивом картинок виселицы. 

- Не забывай, что константы всегда `static final`
```java
public static List<String> STAGES = List.of(...);

//ПРАВИЛЬНО:
public static final List<String> STAGES = List.of(...);
```

- Фактически это константный класс.

В этом классе хранится публичная константа c картинками.  
Для ООП *здесь* это слабовато.

Сделай константу приватной и добавь метод получения картинки:
```java
//БЫЛО:
public class GallowsGraphics {
  //...
  public static List<String> STAGES = List.of(...);
}

//СТАНЕТ:
public class GallowsGraphics {
  //...
  private static List<String> STAGES = List.of(...);

  public static String get(int pictureNumber) {...}
}
```
Теперь этот класс станет более осознанной сущностью.
