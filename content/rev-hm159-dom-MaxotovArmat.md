**4. Не делай сложных условий в if'ах, это плохо читается**

Здесь условие очень трудно понять, потому что в теле if'a выполняется несколько операций.  
Сложные условия объясняй через поясняющие переменные или вспомогательные методы 
```java
if (letter.equals(String.valueOf(word.charAt(i)))) {...}

//ПРАВИЛЬНО (поясняющая переменная):
boolean isВсёОбъясняет = letter.equals(String.valueOf(word.charAt(i)));
if (isВсёОбъясняет) {...}

//ПРАВИЛЬНО (вспомогательный метод):
if (isВсёОбъясняет(...)) {...}

private static boolean isВсёОбъясняет(...) {
  return letter.equals(String.valueOf(word.charAt(i)));
}
```

**5. Не бросай исключения в космос**

Метод main- последняя точка в программе, после которой исключение аварийно закрывает работу программы.  
Если ты знаешь, что в программе могут возникнуть исключения, то обработай их внутри программы
```java
public static void main(String[] args) throws FileNotFoundException
```
Хотя бы просто перехвати, напечатай сообщение "Критическая ошибка, работа программы будет прекращена" и корректно закрой программу.  
Сообщение без конкретики- не самый лучший вариант.  
Но это лучше, чем если программа вылетит с красными эксепшенами и непонятными сообщениями.

- Компилируй регулярные выражения. 

Если регулярное выражение используется многократно, как здесь, то нужно его откомпилировать.  
Компиляция регулярного выражения в объект `Pattern` с помощью `Pattern.compile()` позволяет повысить производительность при многократном использовании
```java
if (letter.matches("[а-яА-ЯёЁ]")) {...}

//ПРАВИЛЬНО:
private static final String REGEX = "[а-яА-ЯёЁ]";
private static final Pattern PATTERN = Pattern.compile(REGEX);

//...
Matcher matcher = PATTERN.matcher(letter);
if(!matcher.matches()) {...}
```
Блох пишет:
```java
"Хотя String.matches — простейший способ проверки, соответствует ли строка регулярному выражению, 
он не подходит для многократного использования в ситуациях, критичных в смысле производительности." 
```
Здесь, конечно, нет критической ситуации с производительностью.  
Но здесь нет и необходимости просто так использовать неоптимальный алгоритм при работе с регулярными выражениями.

**6. Рекурсия**

В алгоритме программы есть рекурсия
```java
+-> void startGameRound() 
|         ↓
|        ...
|         ↓
|   void startGameLoop() 
|         ↓
+---void restartGame() { ... startGameRound();}
```

Рекурсия должна использоваться только там, где она оптимальна алгоритмически.
Например, при обходе бинарного дерева.

В других ситуациях, где рекурсия может быть замененена циклом, она должна быть им замененена.  
Рекурсия тяжело читается и при определенных условиях может привести к переполнению стека.  
Убери рекурсию, замени ее использование на цикл while(). 

Условный пример
```java
//❌ ПЛОХО:
void start() {
  printMessage("Введите команду: 0-выйти, 1-играть");  
  int command = inputCommand();

  if(command == 0) {
    return;
  } 

  if(command == 1) {
    playGame();
  } 

  start();  <-- РЕКУРСИЯ
}

//✔️ ХОРОШО:
void start() {
  while(true) {
    printMessage("Введите команду: 0-выйти, 1-играть");  
    int command = inputCommand();

    if(command == 0) {
      return;
    } 

    if(command == 1) {
      playGame();
    } 
  }  
}
```

- Аргументы в методах.

Р.Мартин утверждает, что идеальный метод это тот, который ничего не принимает и ничего не возвращает.  
Но на практике это не всегда так. Иногда входящие аргументы поясняют те действия, которые выполняет метод. 

У тебя тут есть методы, которые ничего не принимают, хотя стоило бы.

Например метод, который добавляет букву в список введенных.  
Я бы ожидал, что он в себя примет букву, которую нужно ввести в список введенных- это логично и естественно
```java
void addLetterToEnteredLettersArray()

//ПРАВИЛЬНО:
void addEnteredLetter(chat letter)
```

Если открываем букву в слове, то хотелось бы более явно видеть, откуда она берется:
```java
void updateCurrentWord() {
  //...  
  if (letter.equals(String.valueOf(word.charAt(i)))) {
    CURRENT_WORD.setCharAt(i, word.charAt(i));
  }
  //...
}

//ПРАВИЛЬНО:
void openLetter(char letter) {
  //...  
  if (word.charAt(i) == letter) {
    CURRENT_WORD.setCharAt(i, letter);
  }
  //...
}
```
В этом случае команда будет звучать "открой вот эту букву".  
А не "открой ту букву, которую мы когда-то при каких-то обстоятельствах записали в окружение".
