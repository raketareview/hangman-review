**2. Текст в exception всегда должен быть на английском языке**
```java
throw new IllegalStateException("Список слов пуст. Проверьте ресурс: " + wordsResource);
```
Исключение это не просто телеграмма, которая летит сквозь слои. 
У exception особое назначение- если исключение вылетит и не будет перехвачено внутри программы, то аварийно прекратит выполнение программы.
Тогда на экране будет распечатано сообщение эксепшена, и это сообщение должно быть понятно сисадмину в любой точке планеты. 
А значит, сообщение должно быть на английском. 
Интерпритация исключения и перевод его на локальный язык должно происходить там, где это соответствует архитектуре программы.  
Или не происходить вовсе, если исключение не планируется перехватывать.

Лучше файл со словами прочитать один раз- при вызове конструктора

```java
public String getWord() {

```

- Реакция на ошибки.

Если по указанному адресу в `class Dictionary` не будет найден файл со словами, 
то программа аварийно вылетит и напечатает непонятные для юзера сообщения в консоль
```java
1. Начать новую игру
2. Выход
1
Exception in thread "main" java.lang.IllegalStateException: Ресурс не найден: data.txt_

	at hangman.word.TxtWordProvider.getWord(TxtWordProvider.java:28)
```
Здесь должна быть другая реакция на отсутствие файла со словами.  
Нужно сказать юзеру, что файл со словами по указанному пути открыть не удалось и поэтому работа программы будет завершена. 

Причем, путь нужно указать не относительный, как у тебя: `data.txt_`.  
А абсолютный, чтобы юзер знал, где конкретно ему нужно искать файл: `c:/..../data.txt_`.

**6. class ConsoleMenu**

- Меню в процедурном стиле. Для простой программы- почему бы и нет.  
Про меню в ООП стиле я писал [ТУТ](https://t.me/zhukovsd_it_chat/53243/114908).

- Нарушение SRP, чужая ответственность.  
Класс меню при выборе соответствующего пункта может запустить игру, но не более.
То есть, максимум- это вызвать метод типа `game.start()`.  
Здесь меню не только запускает игру, но и управляет ее ходом в `void playGame()` и `processTurn(Game game)`.

- Спагетти код, лишние связи между классами.

Меню должно использовать Game одним конкретным способом- просто запустить игру.  
Меню не должно обращаться в Game по другим вопросам. 
Например не должно использовать из `Game` методы, которые являются частью внутреней организации класса Game
```java
if (game.acceptsInput(c)) {
  return c;
}
```
Опять же, это сложное переплетение между двумя классами, известное как "спагетти код", 
вызвано неправильным разделением ответственностей между классами `ConsoleMenu` и `Game`.  
Когда ты из меню уберешь управление ходом игры, а оставишь только запуск игры, этот(и подобный ему) спагетти код сам собой рассосется.  

Но если рассмотреть ситуацию в вакууме, то в том случае, когда два равноценных класса используют в своей работе один и тот же алгоритм, 
этот алгоритм нужно вынести в третий класс.  
Например, сейчас `ConsoleMenu` и `Game` используют валидацию русской буквы. Просто вынеси ее в отдельный класс. 
Можешь даже через интерфейс, как ты любишь. Например:
```java
public interface LetterValidation {
  boolean isValid(char letter);
}

public class RussianLetterValidation implements LetterValidation {...}
```

## АРХИТЕКТУРА

Программа написана в ООП стиле.  
Само по себе это уже хорошо.  
Но ООП тут имеет некоторые недостатки.

Хорошо с точки зрения ООП: 
+ Программа разделена на классы, каждый из которых описывает отдельную сущность: движок игры, словарь, вью и т.д.
+ Отдельный класс с точкой входа main, который только конструирует и запускает систему(через класс меню).
+ Используются интерфейсы, что через полиморфизм их реализаций создает точки расширения в программе.

Плохо с точки зрения ООП:
- Неправильное распределение ответственностей между классами, например, между игрой и меню.
- Не выявлены некоторые сущности. Например, напрашивается отдельный класс контроллер игры, функцию которого сейчас в том числе выполняет меню.
- Неправильно декомпозированы некоторые классы- все, которые касаются "слотов".

Ты правильно понял, что в программе есть менее очевидные, чем словарь, сущности.  
Например, слоты. Группа слотов составляет отдельную сущность, которую ты назвал `WordProgress`.  
Уже за это программой засчитать объектно-ориентированной.

Слоты это что-то вроде секретных букв, а `WordProgress` это секретное слово.

Но к сущности секретного слова относится не только набор секретных букв, но и методы работы с ними.  
Например, открыть букву.
По крайне мере, если у нас архитектура программы с богатой моделью(rich model), а всё в проекте указывает именно на это.

По слотам(они же в моей интерпритации "секретные буквы") тоже вопросы- они сделаны так сложно и вычурно, что намерения читаются далеко не сразу.

Я бы их интерпритировал по-другому.  
В секретном слове есть секретные буквы(ладно, пусть они будут "слоты").  
Слот по-любому содержит букву, но юзер ее может видеть, а может не видеть.  
Поэтому кроме буквы слот содержит еще информацию о том, открыта буква или нет, а так же метод ее открытия.  
Поэтому вместо кучи классов и интерфейсов я бы сделал один класс типа такого:
```java
public class Slot{
  private static final boolean CLOSE = false;
  private static final boolean OPEN = true;

  private final char letter;
  private final boolean isOpen;

  //клиентом через публичный конструктор слот всегда создается с закрытой буквой
  public Slot(char letter) {
    this(letter, CLOSE);
  }

  private Slot(char letter, boolean isOpen) {
    this.letter = letter;
    this.isOpen = isOpen;
  }

  //слот с открытой буквой создается только через фабричный метод 
  public Slot open() {
    return new Slot(letter, OPEN);
  }

  public char getLetter() {
    return letter;
  }

  public boolean isOpen() {
    return isOpen;
  }
}
```
В этом случае, так же как в твоих слотах-record'ах, класс остается иммутабельным.

Если иммутабельность не принципиальна, все еще проще:
```java
public class Slot{
  private final char letter;
  private boolean isOpen;

  public Slot(char letter) {
    this.letter = letter;
    this.isOpen = false;
  }

  public void open() {
    isOpen = true;
  }

  public char getLetter() {
    return letter;
  }

  public boolean isOpen() {
    return isOpen;
  }
}
```

Пример хорошей правильной объектно-ориентированной декомпозиции Виселицы на классы можешь посмотреть у Сергея в расширенных материалах.
